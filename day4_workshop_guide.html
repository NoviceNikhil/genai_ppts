LLM APIs, Streaming & Production Patterns

Welcome to Day 4. Today we shift from prompt tweaking to robust infrastructure. We transition the playground models into highly-observable, error-resilient backend services built to handle real scale.

The Paradigm Shift Playground vs Infrastructure

"Playground is for experimenting. Production APIs are infrastructure."

The Playground

  • Click-and-test interface without written code.
  • Great for early prompt exploration & validation.
  • Stateless — zero data persistence between page loads.
  • Shared rate limits across all basic users.

The Production API

  • Programmatic HTTP connection sending JSON payloads.
  • Secret keys loaded strictly via environmental variables.
  • Full parameters control (Temperature, Top-p, Stop sequence).
  • Robust error handling, backoffs, streaming SSE events.

The Messages Array Mental Model

Unlike standard REST requests, every LLM API call is a structured, state-free conversation. The model has absolutely no memory. Your backend wrapper is fully responsible for keeping, ordering, and pruning the historical conversation stack before dispatching the request.

◈

system

Persistent directives shaping model personality, formats, constraints, and runtime rules. Transmitted once at top-level, active throughout the session. Never leaked to clients.

◉

user

Inputs generated directly by the human operator. Can incorporate textual queries, structured documents, dataset tables, or media files that demand programmatic processing.

◇

assistant

The cumulative history of responses returned by the model. Appended sequentially in the message stack to guide conversational flow and continuity.

02 · Module 1

API Foundations & Message Specifications

Understand structural architecture, exact parameter tuning, and comparative multi-provider request structures.

Anthropic vs OpenAI Payload Structures

While both SDKs accept the conversational Messages pattern, their native structures diverge slightly, especially regarding how system contexts are handled.

Anthropic Architecture
  • Endpoint: POST /v1/messages
  • System settings are passed as a **top-level parameter**, outside the messages array.
  • Versioned pinning via custom HTTP headers is mandatory: anthropic-version: 2023-06-01.
  • The payload explicitly requires both model and max_tokens to execute.
OpenAI Architecture
  • Endpoint: POST /v1/chat/completions
  • System settings are packaged as a **standard message object** at index 0 of the messages array.
  • Alternative parameters such as response_format: {"type": "json_object"} are used to force structured returns.
  • Parameters are lighter, utilizing automatic internal server-side bounds when omitted.

Key API Parameters & Fine-Tuning

Adjust parameters dynamically inside backend handlers to balance performance, cost, and creative focus.

🌡️

Temperature (0.0 - 1.0)

Low (0.0) yields deterministic, focused outputs suitable for code and structured extraction. High (0.8+) increases randomness and linguistic variation.

🎯

Top-P (Nucleus Sampling)

Defines search selection criteria. A setting of 0.9 limits candidate generation to the top 90% probability spectrum. Do not adjust concurrently with Temperature.

🛑

Stop Sequences

Array of terminal strings. Reaching any match halts generation instantly. A common example is using ["}"] to terminate JSON extractions cleanly.

03 · Module 2

SDK Engineering & Client Security

API Key containment patterns, thin backend architecture, and threat vectors.

The Security Threat Model

Shipping keys inside client-side JS bundles constitutes an instant credential leak.

⚠️

The Browser Vulnerability

Any browser code can be inspected. Network tabs log request headers directly. Bundles are easily parsed, exposing keys instantly. There is no such thing as frontend obfuscation.

🛡️

The Backend Shield Pattern

Keys reside strictly inside memory environment scopes on a secure micro-service (e.g., FastAPI, Next.js server actions). The frontend client only requests local endpoints.

The Request Lifecycle Pattern

Secure request orchestration flow in a production application:

01 Frontend Trigger User submits prompt to the custom endpoint /api/chat.
→
02 Backend Gateway FastAPI interceptor authorizes request & appends rate-limiting tokens.
→
03 Secure Proxy Backend loads credentials & dispatches API POST to the provider's server.
→
04 Response return Raw payload parsed. Usage statistics logged. Clean results sent to React.

04 · Module 3

Streaming Systems & SSE Protocols

Deep dive into Server-Sent Events, packet buffering states, and incremental JSON parsing architectures.

Buffering vs Streaming User Experience

Comparing perceived response time metrics between standard POST calls and continuous chunked delivery.

Buffered Delivery
  • Wait time matches completion time (typically 8–15s).
  • Causes higher user friction; spinners promote drop-offs.
  • Prone to connection drops on lengthy requests.
Streaming Delivery (SSE)
  • First token visible in under 200–500ms (TTFT).
  • Users read incrementally while generation proceeds.
  • Significantly higher engagement metrics.

The SSE Protocol Standard

Server-Sent Events establish a persistent, unidirectional text pipe over standard HTTP. Response payloads are partitioned into frames separated by double newlines:

HTTP Event Stream Protocol 
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
Transfer-Encoding: chunked

data: {"token": "Hello"}

data: {"token": " world"}

data: [DONE]

Incremental JSON Parser Pattern (The Streamed Tool Call Challenge)

When tools execute via streams, JSON arguments are delivered in fragmented text deltas: {"city": "Lon → don", "temp": 2 → 2}. Standard JSON parsers (json.loads) crash instantly when called on these partial structures.

The Double Buffering Pattern

1. Instantiate a designated string accumulator buffer per active tool_use_id.
2. Append incoming input_json_delta fragments systematically to the matching buffer.
3. Listen for the formal content_block_stop event or the ultimate message_stop before executing json.loads() on the fully assembled buffer.

Exercise 3: Asynchronous FastAPI Token Streamer

This script handles live connections, proxies the AI response stream, and catches unexpected client aborts cleanly.

streamer.py (FastAPI Endpoint) 
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse
from anthropic import AsyncAnthropic
import asyncio
import json

app = FastAPI()
client = AsyncAnthropic()

async def token_generator(prompt: str, request: Request):
    try:
        async with client.messages.stream(
            model="claude-3-5-sonnet-20241022",
            max_tokens=1024,
            messages=[{"role": "user", "content": prompt}]
        ) as stream:
            async for text in stream.text_stream:
                # Check if the frontend disconnected mid-stream
                if await request.is_disconnected():
                    print("Client disconnected. Aborting generation.")
                    break
                
                payload = {"token": text}
                yield f"data: {json.dumps(payload)}\n\n"
            
            yield "data: [DONE]\n\n"
    except Exception as e:
        err_payload = {"error": str(e)}
        yield f"event: error\ndata: {json.dumps(err_payload)}\n\n"

@app.post("/api/stream")
async def stream_chat(body: dict, request: Request):
    generator = token_generator(body["prompt"], request)
    return StreamingResponse(
        generator, 
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}
    )

Function Calling & Orchestration Loops

Bridge LLM reasoning with actual system runtime executions using strict JSON schemas and recursive decision state loops.

The Tool-Calling Mental Model

Models never run arbitrary code blocks directly. They produce structural intent parameters matching predefined schemas. Your application runtime retains the execution contract.

Function Calling Flowchart
01 Schema Dispatch Provide list of tools & JSON parameters descriptions to the API request.
→
02 Model Interception Model generates structured tool_use block matching arguments specs.
→
03 Local Execution Your backend interceptor parses JSON, fires local function, and fetches results.
→
04 Conversational Loop Submit results as a tool_result message type. Model decides next path.

Exercise 4: Implementing Multi-turn Function Orchestration

This complete code implements an orchestrator that validates arguments dynamically and supports self-correction loops.

orchestrator.py 
import json
from anthropic import Anthropic

client = Anthropic()

# 1. Define actual function
def get_database_user(user_id: int):
    db = {101: {"name": "Anant", "tier": "admin"}, 102: {"name": "Sarah", "tier": "pro"}}
    return db.get(user_id, {"error": "User not found"})

# 2. Tool definition schema
USER_TOOL_SCHEMA = {
    "name": "query_user_db",
    "description": "Queries the production DB to fetch user permissions.",
    "input_schema": {
        "type": "object",
        "properties": {
            "user_id": {"type": "integer", "description": "Integer database primary key identifier"}
        },
        "required": ["user_id"]
    }
}

async def agent_loop(user_prompt: str):
    messages = [{"role": "user", "content": user_prompt}]
    max_turns = 5
    
    for turn in range(max_turns):
        resp = client.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=1024,
            tools=[USER_TOOL_SCHEMA],
            messages=messages
        )
        
        # If the model completes reasoning and wants to reply
        if resp.stop_reason == "end_turn":
            return resp.content[0].text
            
        # Intercept tool call
        if resp.stop_reason == "tool_use":
            tool_use_block = [c for c in resp.content if c.type == "tool_use"][0]
            tool_name = tool_use_block.name
            tool_args = tool_use_block.input
            tool_call_id = tool_use_block.id
            
            # Append assistant's intent call to conversation stack
            messages.append({"role": "assistant", "content": resp.content})
            
            # Execute safe wrapper
            if tool_name == "query_user_db":
                try:
                    uid = int(tool_args.get("user_id"))
                    result_data = get_database_user(uid)
                except Exception as ex:
                    result_data = {"error": str(ex), "is_validation_failure": True}
            else:
                result_data = {"error": "Unknown tool."}
                
            # Return results back to the model
            messages.append({
                "role": "user",
                "content": [{
                    "type": "tool_result",
                    "tool_use_id": tool_call_id,
                    "content": json.dumps(result_data)
                }]
            })
06 · Module 5

Reliability Engineering & Fallback Architectures

Designing wrapper resilience using Exponential Backoff, jitter, circuit breakers, and programmatic models mapping.

The Resilience Checklist

Ensure complete protection against transient cloud outages.

⚡

Exponential Jitter Backoff

For standard rate limiting (429), calculate double retries: t = base * (2^attempt) + random_jitter. Prevents server synchronization patterns.

⏱️

Connect/Read Timeouts

Never allow connection calls to hang indefinitely. Bind explicit limits (e.g., connect timeout 3s, read 30s) to keep workers clear.

🔄

Multi-Model Fallback Chain

On primary API structural fail (5xx or timeout limits), immediately downgrade and request alternative providers: Sonnet → Haiku → Cache.

Exercise 5: Building a Reliable Model Client Wrapper

This client implements multi-model fallback, retry backoffs, and timeout triggers.

reliability.py 
import time
import random
from anthropic import Anthropic, APIConnectionError, APITimeoutError, APIStatusError

client = Anthropic()

def resilient_create_message(prompt: str):
    models = ["claude-3-5-sonnet-20241022", "claude-3-haiku-20240307"]
    max_retries = 3
    
    for model in models:
        print(f"Attempting model execution: {model}")
        
        for attempt in range(max_retries):
            try:
                # Bind explicit timeout
                resp = client.messages.create(
                    model=model,
                    max_tokens=512,
                    timeout=10.0, # Halt if API hangs >10 seconds
                    messages=[{"role": "user", "content": prompt}]
                )
                return resp.content[0].text
                
            except (APIConnectionError, APITimeoutError) as transient_err:
                if attempt == max_retries - 1:
                    print(f"Failed transient recovery on model {model}")
                    break # Break out to trigger next model in fallback list
                
                # Exponential backoff + jitter math
                sleep_duration = (2 ** attempt) + random.uniform(0, 0.5)
                print(f"Transient error: {transient_err}. Retrying in {sleep_duration:.2f}s...")
                time.sleep(sleep_duration)
                
            except APIStatusError as status_err:
                # Do not retry on bad request logic errors (400, 401, 403)
                if status_err.status_code in [400, 401, 403]:
                    print(f"Critical client error: {status_err}. Terminating.")
                    raise status_err
                
                # Retry on 5xx or server rate-limits (429)
                sleep_duration = (2 ** attempt) + random.uniform(0, 0.5)
                time.sleep(sleep_duration)
                
    raise RuntimeError("All available models and recovery fallback nodes failed.")

The Sliding Context Window Truncation Strategy

As conversations grow, the system prompt and historical context carry a compounding token cost. Implement a sliding window truncation pattern:

window_manager.py 
def prune_conversation_history(messages: list, max_turns: int = 6):
    # Always preserve the system prompt, but prune oldest user/assistant pairs
    if len(messages) > max_turns * 2:
        pruned = messages[-(max_turns * 2):]
        return pruned
    return messages

The Complete React Streaming Consumer

This implementation decodes the SSE format in real-time, manages state updates, and handles user cancellation gracefully.

ChatComponent.tsx 
import React, { useState, useRef } from 'react';

export const ChatComponent = () => {
    const [messages, setMessages] = useState<{role: string, text: string}[]>([]);
    const [input, setInput] = useState('');
    const [isStreaming, setIsStreaming] = useState(false);
    const abortControllerRef = useRef<AbortController | null>(null);

    const handleSend = async () => {
        if (!input.trim()) return;
        
        setIsStreaming(true);
        const userPrompt = input;
        setInput('');
        
        setMessages(prev => [...prev, { role: 'user', text: userPrompt }, { role: 'assistant', text: '' }]);

        // Create new AbortController
        const controller = new AbortController();
        abortControllerRef.current = controller;

        try {
            const res = await fetch('/api/stream', {
                method: 'POST',
                headers: { 'Content-Type': 'application/json' },
                body: JSON.stringify({ prompt: userPrompt }),
                signal: controller.signal
            });

            const reader = res.body?.getReader();
            const decoder = new TextDecoder();
            if (!reader) throw new Error('No reader available');

            while (true) {
                const { done, value } = await reader.read();
                if (done) break;

                const chunk = decoder.decode(value);
                const lines = chunk.split('\n');
                
                for (const line of lines) {
                    if (line.startsWith('data: ')) {
                        const dataStr = line.slice(6);
                        if (dataStr.trim() === '[DONE]') break;
                        
                        const payload = JSON.parse(dataStr);
                        setMessages(prev => {
                            const updated = [...prev];
                            const last = updated[updated.length - 1];
                            last.text += payload.token;
                            return updated;
                        });
                    }
                }
            }
        } catch (err) {
            if (err.name === 'AbortError') {
                console.log('Stream manually aborted by user.');
            } else {
                console.error('Stream error:', err);
            }
        } finally {
            setIsStreaming(false);
            abortControllerRef.current = null;
        }
    };

    const handleStop = () => {
        abortControllerRef.current?.abort();
    };

    return (
        <div className="chat-container">
            <div className="messages-log">
                {messages.map((m, i) => (
                    <div key={i} className={`message ${m.role}`}>
                        <b>{m.role}:</b> {m.text}
                    </div>
                ))}
            </div>
            <input value={input} onChange={e => setInput(e.target.value)} disabled={isStreaming} />
            <button onClick={handleSend} disabled={isStreaming}>Send</button>
            {isStreaming && <button onClick={handleStop}>Stop</button>}
        </div>
    );
};

1.3 · May 2026 Reality Check: AI-Augmented Backend Engineering

May 2026 Reality Check: AI-Augmented Backend Engineering

The goal today is not to prove that AI can type backend code. By May 2026, every serious coding assistant can scaffold a basic CRUD API. The valuable skill is knowing how to turn that speed into a reliable backend without accepting fragile architecture, unsafe migrations, inconsistent errors, or insecure auth code.

The useful mental model is:

  • AI accelerates implementation: boilerplate, route files, schema conversions, migration drafts, test drafts, OpenAPI annotations, refactors across multiple files.

  • The bottleneck has shifted: syntax is no longer the slowest part. Context quality, constraints, verification discipline, and refinement loops are now the hard parts.

Today's repeated pattern is:

text

If learners remember only one principle, make it this: AI-generated backend code is a fast first draft, not an engineering decision.

Traditional vs AI-Native Backend Flow

Most engineers learn a traditional flow that looks like this:

The AI-native flow in 2026 is meaningfully different:

The difference is not "AI does it for you." The difference is that implementation speed is no longer the bottleneck. Context quality, architectural judgment, and verification discipline are now the hard parts.

What AI Is Strong At for Backend Work

  • CRUD scaffolding and repetitive boilerplate
  • Route generation from specs
  • Schema-to-model conversion

  • Test generation for happy paths

1.8 · 1. FastAPI Scaffolding with Claude Code

1. FastAPI Scaffolding with Claude Code

1.8.1 · What It Is

What It Is

FastAPI scaffolding means generating the initial project structure — directory layout, main application file, route files, database configuration, dependency injection setup, middleware, and configuration management — from a description of what you want to build. Claude Code does this at the project level: it reads your entire directory, plans changes across multiple files, creates them, and can immediately run tests to verify the result.

1.8.2 · Why It Matters for Product Engineers

Why It Matters for Product Engineers

Backend scaffolding is the most time-consuming part of starting a new feature. You know the pattern — it is the same every time — but typing out the boilerplate takes 30–60 minutes and a small wiring mistake (missing dependency injection, import path typo, incorrect async session setup) costs an hour of debugging later. AI scaffolding eliminates the typo category of bugs and lets you focus on business logic.

1.8.3 · What Learners Must Understand in Depth

What Learners Must Understand in Depth

FastAPI scaffolding is not just creating main.py. A useful backend scaffold has explicit layers:

  • Routes define HTTP contracts: paths, methods, status codes, dependencies, response models, and OpenAPI metadata.
  • Schemas define API input/output contracts with Pydantic. These are not the same as database models.
  • Models define persistent data with SQLAlchemy. These should be typed, constrained, and migration-friendly.
  • Services hold business logic. Routes should stay thin so they are easy to test and document.
  • Repositories or data-access helpers isolate database queries when the project grows beyond one resource.
  • Dependencies provide database sessions, auth context, configuration, and reusable request-scoped objects.
  • Middleware handles cross-cutting behavior like CORS, request IDs, logging, and timing.
  • Core configuration centralizes environment variables, secrets, app metadata, and database URLs.

AI tools are very good at producing this shape if you name the layers. If you only ask for "a FastAPI backend," the assistant may generate a single large file, sync database sessions inside async routes, duplicated enums, or route functions that contain validation, business logic, and database writes all together.

How to Do It

Step 1: Create a CLAUDE.md file. This is the single most important thing you can do before prompting Claude Code for backend work. Without it, you get generic boilerplate. With it, you get code that matches your team's conventions.

Create a CLAUDE.md in your project root:

markdown
# Project: User Feedback Service

## Stack
- Python 3.12, FastAPI 0.115+, SQLAlchemy 2.0 (async), Pydantic v2
- PostgreSQL 16, Alembic for migrations
- uv for package management

## Conventions
- Use `Mapped[T]` type annotations for all SQLAlchemy models
- Separate Pydantic schemas from SQLAlchemy models (schemas/ vs models/ directories)
- Use async sessions everywhere — `AsyncSession` + `create_async_engine`
- Service layer pattern: routes call services, services call repositories
- All routes return structured JSON error responses, never raw exceptions
- Use `Field()` for Pydantic validation with descriptions
- Snake_case for Python, camelCase only in JSON responses (use Pydantic `alias_generator`)

## Directory Structure
app/
  main.py
  core/
    config.py      # Settings via pydantic-settings
    database.py    # Engine, session factory, Base
    security.py    # JWT utilities

- Global exception handler returns {detail: str, error_code: str, status_code: int}

1.8.6 · What Can Go Wrong

What Can Go Wrong

Generic boilerplate without CLAUDE.md. Without it, Claude Code produces functional but generic code — synchronous SQLAlchemy, no service layer, everything in one file. CLAUDE.md communicates your team's standards.

Outdated patterns. Without specifying SQLAlchemy 2.0 explicitly, you might get 1.4 patterns (Column() instead of Mapped[T]). Be explicit about versions in CLAUDE.md.

Over-engineering on the first pass. Claude Code might add caching, rate limiting, and monitoring. Constrain your prompt: "Create the minimum viable scaffold. No caching, no background tasks. We will add those later."

Not reviewing the generated code. The scaffold will compile and health checks will pass, but you must read every file. You are the engineer. The AI is the typist.

Circular imports. AI frequently wires models, schemas, services, and routes in a way that imports the app back into itself. If the server fails at startup with import loops, ask Claude Code to explain the dependency graph before it edits.

Async/sync mixing. In 2026 this is still a common generated-code bug: an async def route calls a synchronous SQLAlchemy Session, or a sync function returns an un-awaited coroutine. Search for Session( and await during review.

Package drift. AI may invent old or non-existent package names. Verify the generated dependency file and pin only packages the app actually imports.

1.9 · 2. Database Schema Design with AI

2. Database Schema Design with AI

1.9.1 · What It Is

What It Is

Database schema design is the process of defining your tables, columns, types, relationships, indexes, and constraints. AI assistance here means describing your data domain in plain language and getting back a normalized schema with appropriate indexes, foreign keys, and migration scripts.

1.9.2 · Why It Matters

Why It Matters

A bad schema is the most expensive technical debt in a product. You can refactor code in a day. Refactoring a schema with millions of rows and active queries takes weeks and carries production risk. Getting the schema right early saves enormous pain later.

1.9.3 · What Learners Must Understand in Depth

What Learners Must Understand in Depth

AI can draft a schema, but it cannot infer the real product workload unless you tell it. Schema design starts from business rules and access patterns:

  • Entities: feedback item, submitter, product area, comments, attachments, status history.
  • Relationships: one user can submit many items; one feedback item can have many comments; one item belongs to one product area.
  • Constraints: required fields, allowed statuses, priority range, uniqueness rules, foreign keys, soft-delete behavior.
  • Read patterns: list by status, category, priority, date range, product area, user email, and search term.
  • Write patterns: create feedback, update status, edit user-owned feedback, delete or soft-delete feedback.
  • Scale assumptions: 100K rows behaves differently from 100M rows; read-heavy workloads need different indexes from write-heavy workloads.
1.9.4 · Normalization: Principles and Practical Tradeoffs

Normalization: Principles and Practical Tradeoffs

Teach normalization as a spectrum, not a binary rule:

  • 1NF (First Normal Form): one atomic value per column. Avoid comma-separated categories or JSON blobs for fields you filter often. If tags is a JSON array and you query WHERE 'bug' = ANY(tags), you lose index support.
  • 2NF (Second Normal Form): every non-key attribute depends on the entire primary key. This matters most when composite primary keys appear. A feedback_category table where category_name depends only on category_id, not on feedback_id, is 2NF compliant.
  • 3NF (Third Normal Form): avoid transitive dependencies — storing product_area_name in the feedback row when you already have product_area_id that joins to the name. Update the name once, not across millions of rows.
  • Intentional denormalization: acceptable when a read path is hot and the duplicated value has a clear update strategy. Storing user_email directly on Feedback rather than a foreign key to a Users table is a deliberate choice for a read-heavy, user-email-keyed access pattern.

In May 2026, the most common AI schema failures are:

  • Over-normalizing into too many tables (7 tables where 3 suffice)
  • Using JSON columns for fields you filter on
  • Missing indexes for filtered list endpoints
  • Omitting foreign key behavior (ON DELETE CASCADE vs RESTRICT)
  • Generating migrations that work forward but fail backward

The SQLAlchemy 2.0 model Claude Code should produce:

The model should use Mapped[T] annotations — this is the SQLAlchemy 2.0 pattern that gives you full IDE type-checking support. It should look something like this structure:

python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column
from sqlalchemy import String, Integer, DateTime, func, Index
from uuid import UUID, uuid4
from datetime import datetime
from enum import Enum as PyEnum

class Base(DeclarativeBase):
    pass

class FeedbackStatus(str, PyEnum):
    OPEN = "open"
    IN_PROGRESS = "in_progress"
    RESOLVED = "resolved"
    CLOSED = "closed"

class FeedbackCategory(str, PyEnum):
    BUG = "bug"
    FEATURE = "feature"
    IMPROVEMENT = "improvement"

class Feedback(Base):
    __tablename__ = "feedback"

    id: Mapped[UUID] = mapped_column(primary_key=True, default=uuid4)
    user_email: Mapped[str] = mapped_column(String(255), index=True)
    category: Mapped[FeedbackCategory]
    title: Mapped[str] = mapped_column(String(500))
    description: Mapped[str]
    priority: Mapped[int] = mapped_column(Integer, default=3)
    status: Mapped[FeedbackStatus] = mapped_column(default=FeedbackStatus.OPEN)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now(), onupdate=func.now()
    )

    __table_args__ = (
        Index("ix_feedback_status_priority", "status", "priority"),
        Index("ix_feedback_category_created", "category", "created_at"),
    )

Key decisions to verify in the generated schema:

  • UUID primary keys instead of auto-increment integers. UUIDs do not leak information about table size and work better in distributed systems.
  • Composite indexes matching your query patterns. If you always filter by status AND priority, a composite index is significantly faster than two single-column indexes.
  • timezone=True on DateTime columns. Always store timestamps with timezone. This is nearly impossible to fix retroactively.
  • String length limits. String(255) for emails, String(500) for titles — not unlimited Text everywhere.
  • Check constraints for values like priority 1-5. Pydantic validates the request, but the database should still protect itself.
  • Server-side defaults for timestamps and status where appropriate, so non-API writes do not create invalid rows.
  • Soft delete decision. If you include deleted_at, every list query must filter it; if you hard delete, audit and recovery are harder.
1.9.6 · Indexing Decisions Learners Should Be Able to Explain

Indexing Decisions Learners Should Be Able to Explain

An index is not "make this faster" magic. It is a tradeoff: faster reads for matching queries, slower writes, extra storage, and more migration complexity.

For the lab API:

QueryUseful IndexReason
WHERE status = 'open' ORDER BY priority, created_at(status, priority, created_at)Matches common triage list
WHERE user_email = ? ORDER BY created_at DESC(user_email, created_at)Supports user history
WHERE category = ? AND created_at >= ?(category, created_at)Supports category/date filters
WHERE id = ?primary keyAlready indexed

Ask AI to explain index order. If it cannot connect the index to a real query, remove or revisit that index.

  • [ ] Queryable fields placed inside JSON
  • [ ] Missing check constraint for priority

1.9.8 · What Can Go Wrong

What Can Go Wrong

Missing downgrade logic. AI-generated migrations often have thorough upgrade() but downgrade() that just calls op.drop_table() without dropping indexes or enum types first. This fails on rollback.

Enum handling. PostgreSQL enums are database-level types. If upgrade() adds an enum type, downgrade() must drop it explicitly. Many AI-generated migrations miss this.

Over-normalization. The AI might produce 8 tables where 3 suffice. Challenge every join table: "Will I actually query this independently?"

Index bloat. Each index slows INSERT/UPDATE. For 100K rows, negligible. For 10M rows on a frequently-written table, measurable.

1.10 · 3. Request Validation and Error Handling with Pydantic v2

3. Request Validation and Error Handling with Pydantic v2

1.10.1 · What It Is

What It Is

Pydantic v2 is the validation layer between the outside world and your application logic. You define schemas — Python classes with type annotations and validation rules — and Pydantic guarantees that any data reaching your service layer has already been validated, coerced, and structured correctly.

1.10.2 · Why It Matters

Why It Matters

Every production incident caused by bad input data is one that Pydantic could have prevented. When a frontend sends priority: "high" instead of priority: 3, you want a clear 422 response — not a 500 from deep inside your database layer. Pydantic v2 is significantly faster than v1 and has better error messages out of the box.

1.10.3 · What Learners Must Understand in Depth

What Learners Must Understand in Depth

Validation has three layers:

  • API validation: Pydantic rejects malformed external input before it reaches business logic.
  • Domain validation: services enforce business rules like allowed status transitions and ownership.
  • Database validation: constraints protect the data even if a future script or admin tool bypasses the API.

AI-generated backends often blur these layers. A common failure is putting all validation in Pydantic and assuming the database no longer needs constraints. Another common failure is using database exceptions as user-facing API responses. The production pattern is layered: reject bad requests early, raise domain-specific exceptions in services, and keep database errors from leaking implementation details.

For this workshop, learners should distinguish:

  • Request schema: what clients may send, usually strict and minimal.
  • Update schema: what clients may patch or replace, often optional fields with at least-one-field validation.
  • Response schema: what the API promises to return, never every database column.
  • Internal model: how data is stored, including internal fields that may never leave the backend.
  • Error schema: the stable shape clients can handle for failures.

1.10.6 · What Can Go Wrong

What Can Go Wrong

Pydantic v1 syntax. If the AI generates class Config: inside the model instead of model_config = ConfigDict(...), you are looking at Pydantic v1 syntax. Specify "Pydantic v2" explicitly in your prompts and in CLAUDE.md.

Missing EmailStr import. EmailStr requires the email-validator package. If the AI uses it but does not add the dependency, you get an import error at runtime. Check your requirements.

Over-validation on updates. All fields should be optional in FeedbackUpdate. If the AI makes fields required, a client updating only status must send every other field too. Common AI-generated bug.

Enum duplication. You may end up with enums defined independently in both SQLAlchemy models and Pydantic schemas with slightly different values. Put shared enums in app/core/enums.py and import from there.

Inconsistent error shapes. One route returns {"detail": "not found"}, another returns {"message": "not found"}, and validation returns FastAPI's default list. Decide whether to wrap all errors or document where FastAPI defaults remain.

Leaking internals. Generic exception handlers must log the real exception server-side but return a generic client message. AI sometimes returns str(exc), which can expose table names, SQL, secrets, or file paths.

1.11 · 4. OpenAPI / Swagger Spec Generation

4. OpenAPI / Swagger Spec Generation

1.11.1 · What It Is

What It Is

FastAPI automatically generates OpenAPI 3.1 specifications from your route definitions, type annotations, and Pydantic models. This means every endpoint, request body, response schema, query parameter, and error response is documented in a machine-readable format — and FastAPI serves interactive documentation at /docs (Swagger UI) and /redoc (ReDoc) with zero additional configuration.

1.11.2 · Why It Matters

Why It Matters

The OpenAPI spec is not just documentation. It is a contract between your backend and every consumer: frontend team, mobile developers, QA automation, and increasingly, AI agents. A notable 2025-2026 trend is converting OpenAPI specs into MCP servers, allowing AI agents to discover and call your APIs programmatically. Accurate specs make downstream tools work. Vague specs break everything built on top of them.

1.11.3 · What Learners Must Understand in Depth

What Learners Must Understand in Depth

FastAPI generates OpenAPI from code, but only from the information you provide. The framework can infer paths and basic schemas, yet it cannot invent useful examples, ownership rules, domain-specific error codes, or deprecation notes. A generated spec is accurate only when routes, Pydantic models, response models, status codes, and documented error responses are accurate.

In 2026, OpenAPI is used for more than Swagger UI:

  • Frontend SDK generation: typed clients reduce hand-written API mistakes.
  • Postman collections: testers can import the live contract.
  • Contract tests: CI can detect breaking route or schema changes.
  • AI tool contracts: agents can inspect endpoints, parameters, auth requirements, and response schemas.
  • MCP-style integrations: OpenAPI can become the bridge between internal APIs and AI workflows.

The teaching point: OpenAPI quality is backend quality made visible.

4a. API Versioning Strategies

1.12.1 · What It Is

What It Is

API versioning is how you evolve a backend API without breaking existing clients. In 2026, versioning is not optional for any API used by more than one consumer — including your own frontend and any AI agents that have cached your OpenAPI spec.

1.12.2 · Why It Matters

Why It Matters

AI-generated backends rarely include versioning by default. When you need to change a response shape, rename a field, or remove an endpoint, you need a strategy that lets old clients keep working while new clients get the improved version.

1.12.3 · Common Patterns

Common Patterns

URL path versioning (most common for REST APIs):

python
# Prefix all routes with the API version
app.include_router(feedback_router, prefix="/api/v1")
app.include_router(feedback_v2_router, prefix="/api/v2")

This is the pattern used in the lab. It is visible, easy to test, and works cleanly with OpenAPI (each version gets its own spec).

python
# Client sends: API-Version: 2
@router.get("/feedback")
async def list_feedback(
    api_version: str = Header(default="1", alias="API-Version")

    ...

Less visible but avoids URL changes. Harder to test in a browser.

python
@router.get(

1.13 · 5. Authentication and Authorization Patterns

5. Authentication and Authorization Patterns

1.13.1 · What It Is

What It Is

Authentication (authn) verifies who a user is. Authorization (authz) determines what they can do. For backend APIs, this means JWT tokens for authentication and role/permission checks for authorization. AI tools can scaffold the boilerplate, but the security logic must be reviewed by a human.

1.13.2 · Why It Matters

Why It Matters

Auth is the one area where AI-generated code is most dangerous to accept without review. A subtle bug in token validation creates a security vulnerability that compiles, passes tests, and looks correct on casual review. You must understand every line of auth code.

1.13.3 · What Learners Must Understand in Depth

What Learners Must Understand in Depth

Authentication and authorization are separate:

  • Authentication: "Who is making this request?" Usually proven by a bearer token, session cookie, API key, or OAuth provider.
  • Authorization: "What is this identity allowed to do?" Usually enforced through ownership checks, roles, scopes, permissions, or policies.

For today's lab, JWT scaffolding is enough. For production, teams often delegate login and identity to an IdP such as Auth0, Okta, Cognito, Clerk, WorkOS, or an internal platform. Even then, your FastAPI service still owns authorization checks: a valid token does not automatically mean the user can update a specific feedback item.

Teach the request path:

text
HTTP request
  -> Authorization: Bearer <token>
  -> auth dependency validates signature, expiry, issuer, audience
  -> dependency returns current user / principal
  -> route passes principal to service
  -> service checks ownership or permission
  -> service performs write

AI can scaffold this path, but learners must inspect every trust boundary.

1.13.4 · How to Do It

How to Do It

Review checklist for AI-generated auth code:

After Claude Code generates the auth module, check these items manually:

  1. Token secret is from environment variables, not hardcoded in the source
  2. Expiry is validated — the dependency raises 401 if the token is expired
  3. Algorithm is explicit — algorithms=["HS256"] is passed to the decode function, not inferred
  4. Error handling is specific — expired tokens return "Token expired," invalid tokens return "Invalid token," missing headers return "Not authenticated." Not a generic 403 for everything
  5. The dependency does not trust the token payload blindly — if the token says role: admin, the dependency should verify that claim against your database, not just pass it through
  6. Issuer and audience are validated when tokens come from an external identity provider
  7. Password hashing uses a modern password hashing library if local passwords are in scope; never store plaintext or reversible encrypted passwords
  8. Authorization checks happen in the service layer, not only in frontend UI or route decorators
  9. Tests cover forbidden access, not just successful login

1.13.6 · What Can Go Wrong

What Can Go Wrong

Hardcoded secrets. The most common AI-generated auth mistake. If you see SECRET_KEY = "your-secret-key-here" anywhere other than a .env.example file, fix it immediately.

Weak algorithm defaults. Some AI-generated code uses "none" or "HS384" as the algorithm. HS256 is standard for symmetric JWTs. RS256 is better for production (asymmetric, allows public key verification without sharing the signing key). Never accept "none".

Missing token refresh. The AI will scaffold access token creation but often skip refresh tokens entirely. For a training exercise this is fine, but flag it as something you would add before production.

OAuth complexity explosion. If you mention "OAuth" in a prompt, AI tools will generate hundreds of lines covering authorization code flow, PKCE, token exchange, and callback handling. Start with simple JWT. Add OAuth only after the basic flow works.

Confusing authn and authz. AI may validate that a token is real but forget to check that the authenticated user owns the resource being changed.

Frontend-only permissions. A generated frontend may hide buttons for unauthorized users, but the backend must still enforce permissions.

1.14 · 6. SQL Query Generation and Optimization

6. SQL Query Generation and Optimization

1.14.1 · What It Is

What It Is

Using AI to write and optimize SQLAlchemy ORM queries for your product's data access patterns. This is product SQL — "Get all open feedback items for this user, sorted by priority, paginated" — not analytics SQL with complex multi-table joins.

1.14.2 · Why It Matters

Why It Matters

Most product engineers write correct-but-slow queries — full table scans when an index exists, loading 50 columns when the endpoint needs 3, or N+1 queries in a loop. AI tools are good at spotting these patterns if you tell them your indexes and data volume.

1.14.3 · What Learners Must Understand in Depth

What Learners Must Understand in Depth

This module is about product queries, not analytics. Product queries serve user-facing flows and must be predictable under load:

  • list feedback with filters and pagination
  • fetch one feedback item by ID
  • update one item with an ownership check
  • delete or soft-delete one item
  • count open items for a dashboard badge
  • list one user's feedback history

The performance questions are practical:

  • Does the query use the index we designed?
  • Does pagination use LIMIT and OFFSET or cursor-based pagination intentionally?
  • Does count happen in the database, not Python memory?
  • Does the endpoint fetch only the data it returns?
  • Are related rows loaded in batches rather than one query per item?
  • Are writes wrapped in a transaction when multiple rows change together?

AI can write SQLAlchemy syntax quickly. It cannot know real cardinality, skew, or query-plan behavior without telemetry. For anything performance-sensitive, use database tools such as EXPLAIN ANALYZE against realistic data.

1.14.4 · The N+1 Query Problem

The N+1 Query Problem

This is the most common query failure in AI-generated backends. It happens when the code issues one query to fetch a list, then one additional query per item to fetch related data.

python
# Bad: N+1 — fetches all feedback, then one query per item for its category details
feedbacks = await db.execute(select(Feedback))
for fb in feedbacks.scalars().all():
    category = await db.execute(select(Category).where(Category.id == fb.category_id))
    # one query per feedback item
python
# Good: single query with joinedload or selectinload
from sqlalchemy.orm import selectinload

result = await db.execute(
    select(Feedback).options(selectinload(Feedback.category))
)
feedbacks = result.scalars().all()

AI generates N+1 patterns constantly when models have relationships. After any scaffold that involves relationships, ask:

text
Review every query in app/services/ for N+1 patterns.
For any query that loads related objects in a loop, rewrite it using selectinload or joinedload.
Explain why each change is needed.

A common pitfall — the pagination count query:

1.15 · 6a. Rate Limiting and Throttling

6a. Rate Limiting and Throttling

1.15.1 · What It Is

What It Is

Rate limiting controls how many requests a client can make in a given time window. Throttling slows down requests rather than rejecting them outright. Both protect your backend from abuse, scraping, and unintentional denial-of-service from buggy clients.

Why It Matters

1.16 · 6b. Structured Logging and Observability

6b. Structured Logging and Observability

1.16.1 · What It Is

What It Is

Structured logging means emitting log records as JSON objects with consistent fields — timestamp, level, request ID, user, endpoint, duration, status code — rather than free-text strings. Observability means making the system's internal state visible enough to diagnose problems without attaching a debugger.

1.16.2 · Why It Matters

Why It Matters

AI-generated backends ship with print() statements or no logging at all. In production, you cannot debug a 500 error that happened three hours ago without logs. Structured logs make it possible to search, filter, and alert on specific conditions.

1.16.3 · The Logging Pattern

The Logging Pattern

AI almost never adds structured logging unless you ask for it explicitly. When you do ask, use this prompt:

text
Add structured JSON logging to the feedback service using Python's structlog 24.x:
- Every service method should log: method name, input parameters (excluding PII), result type
- Every HTTP request should log: method, path, status_code, duration_ms, request_id
- Log exceptions with stack traces at ERROR level, never at INFO
- Never log email addresses or user-identifiable data in INFO logs
- Output JSON format in production, human-readable format in development (based on ENV variable)

Example structured log output:

json
{
  "timestamp": "2026-05-24T10:30:00Z",
  "level": "info",
  "event": "feedback.created",
  "request_id": "req_01HX...",
  "feedback_id": "2df2fd66-...",
  "category": "bug",
  "priority": 1,
  "duration_ms": 42
}

Adding request ID middleware:

python
# app/middleware/request_id.py
import uuid
from starlette.middleware.base import BaseHTTPMiddleware

class RequestIDMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        request_id = str(uuid.uuid4())
        request.state.request_id = request_id
        response = await call_next(request)
        response.headers["X-Request-ID"] = request_id
        return response

The request ID should appear in every log line and every error response. This is what lets you trace a user complaint ("my request failed at 10:30") to the exact log sequence.

1.16.4 · Observability Principles for AI-Generated Backends

Observability Principles for AI-Generated Backends

  • Log at service boundaries, not inside utility functions
  • Include request_id in every log line using a context variable or structlog's bind_contextvars
  • Measure duration for every external call (database, HTTP, cache)
  • Never log secrets, tokens, or passwords — even accidentally via str(exception)
  • Use different log levels consistently: DEBUG for development detail, INFO for normal operation, WARNING for expected-but-abnormal events, ERROR for failures requiring attention

1.17 · 7. The AI Pair-Programming Rhythm for Backend

7. The AI Pair-Programming Rhythm for Backend

1.17.1 · What It Is

What It Is

A repeatable workflow for building backend features with AI assistance: spec, scaffold, test, refine, ship. This is not "type a prompt and ship the result." It is a structured process where you drive the design decisions and the AI handles the implementation typing.

1.17.2 · Why It Matters

Why It Matters

Without a rhythm, AI-assisted development devolves into two failure modes: prompting once and shipping unreviewed code (dangerous), or spending more time on prompts than writing the code yourself (wasteful). The rhythm is consistently faster than manual coding while maintaining quality.

1.17.3 · What Learners Must Understand in Depth

What Learners Must Understand in Depth

Backend AI work needs a cadence because generated code compounds. If the schema is vague, the models are wrong. If the models are wrong, the migration is wrong. If the migration is wrong, routes and tests may still pass against the wrong data contract. The solution is not one giant perfect prompt; it is a sequence of short, inspectable steps.

Use this loop for every backend feature:

text
1. Spec: human writes the behavior and constraints.
2. Plan: AI proposes file-level changes without editing.
3. Scaffold: AI creates the first implementation.
4. Inspect: human checks architecture, auth, database, and contracts.
5. Test: AI drafts tests; human rejects weak tests.
6. Refine: AI fixes failures and review findings.
7. Verify: run app, tests, migrations, OpenAPI, manual endpoint checks.
8. Ship: commit only when behavior and contracts are understood.

The rhythm also decides which tool to use:

  • Claude Code: plan, scaffold, migrations, multi-file changes, test-fix loops, final repo review.
  • Cursor: inspect code, ask cross-file questions, apply targeted refactors, review diffs, maintain project rules.

  • Git: capture a working checkpoint before the next risky change.
1.17.4 · AI Verification Discipline

AI Verification Discipline

Verification is a core engineering competency in 2026, not an optional step. The common mistakes to call out explicitly:

Never accept migrations blindly. Generated migrations compile and run — that does not mean they are correct. Read the upgrade() function. Verify the downgrade() function reverses it completely. Run the migration on a test database before production.

Never accept auth code blindly. Generated auth code almost always has at least one of: hardcoded secret, missing expiry check, wrong algorithm, or missing issuer validation. Read every line.

Never accept async code blindly. Search for Session( (sync) inside async routes, un-awaited coroutines, and blocking I/O calls inside async functions.

Generated code compiles ≠ correct. The test suite passing is necessary but not sufficient. Read the test assertions. A test that checks response.status_code == 200 without checking the response body is testing FastAPI's HTTP stack, not your logic.

1.17.5 · The Five Steps

The Five Steps

Step 1 — Spec (5 minutes, you write this). Write a plain-text spec before touching Claude Code. Two paragraphs maximum: what the feature does, what endpoints it needs, what data it stores, what the error cases are. If the AI-generated code does not match this spec, the code is wrong.

Step 2 — Scaffold (10 minutes, AI does the typing). Give Claude Code the spec. Use plan mode first for anything touching 3+ files. Review the plan, approve it, let it execute. Note small corrections for the refine step — do not interrupt.

Step 3 — Test (10 minutes, AI generates, you review). Ask Claude Code to generate pytest tests for the scaffolded code:

This is where the learning happens. Failures teach you what the AI got wrong; fixing them teaches you how the framework works.

Step 5 — Ship (5 minutes). Once tests pass, do a final review:

text

- Any security issues? (SQL injection, missing auth, data leaks)
- Any missing error handling?
- Are all async operations properly awaited?
- Does the OpenAPI spec look correct? Start the app and check /docs.

1.18.3 · 8.3 Streaming-First Architecture (SSE vs WebSockets for AI)

8.3 Streaming-First Architecture (SSE vs WebSockets for AI)

Maturity: Mainstream (SSE); Growing (WebSockets for agents)

AI generation is inherently streaming — users see tokens appear rather than waiting for complete responses. This forces a streaming-first architecture decision early. The standard pattern for generation is SSE (Server-Sent Events): client sends a POST, server streams tokens back over a long-lived HTTP connection. SSE is simple, firewall-friendly, and sufficient for one-way token streaming. WebSockets are gaining traction for agentic multi-turn flows where the client needs to send tool results or interrupts mid-stream — the bidirectional channel allows richer agent-to-backend communication. Time to First Token (TTFT) has become a first-class backend performance metric alongside p95 latency and throughput. FastAPI handles both patterns well:

python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio

app = FastAPI()

async def token_generator(prompt: str):
    # Call your LLM here
    async for token in llm_stream(prompt):
        yield f"data: {token}\n\n"
    yield "data: [DONE]\n\n"

@app.post("/api/v1/generate")
async def generate(request: GenerateRequest):
    return StreamingResponse(
        token_generator(request.prompt),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
    )

Choosing the wrong protocol (SSE when you need mid-stream client messages, or WebSockets where SSE suffices) means rebuilding the entire messaging layer later. Decide based on whether the client needs to send data after the stream starts.

1.20 · Key Takeaways

Key Takeaways

  1. AI in 2026 is an accelerator, not an architect. It can scaffold fast, but humans still own product decisions, boundaries, security, and data correctness.
  1. CLAUDE.md and project rules are leverage multipliers. They turn generic codegen into repo-specific codegen.
  1. Schemas, models, routes, and migrations must stay separate. That separation keeps APIs honest and migrations survivable.
  1. OpenAPI is a live contract. If the docs are wrong, downstream SDKs, Postman, and AI agents all inherit the mistake.
  1. Auth deserves adversarial review. Treat every generated auth dependency as guilty until you have checked secrets, expiry, issuer, audience, and ownership rules.
  1. The backend workflow is repeatable: spec, plan, scaffold, inspect, test, refine, verify, ship. That rhythm is the real skill.
  1. Verification is a core competency. Never accept migrations, auth code, or async queries without reading them. Generated code compiling is not the same as generated code being correct.
  1. Rate limiting and logging are not optional. Add them with a Hardening Prompt before shipping. AI will not add them unless you ask.

Day 5: RAG, Embeddings & Semantic Search

Foundation Track · Comprehensive Courseware Guide

RAG, Embeddings & Semantic Search

Search is no longer keyword matching. This is how AI understands meaning.

Embeddings as Semantic Coordinates

  • An embedding is a list of numbers that encodes the meaning of text — not its letters, not its keywords, but its semantic content

  • Words, sentences, and documents with similar meanings produce vectors that are close together in high-dimensional space

  • OpenAI's text-embedding-3-small produces 1536 numbers per input — that's your semantic coordinate

  • Unlike keyword search, embeddings capture synonyms, paraphrases, and conceptual similarity automatically

  • This is the mathematical foundation of every AI search system, recommendation engine, and RAG pipeline you will build

Why Semantic Retrieval Changes Products

◈ Keyword Search Breaks: User types 'cancel account' — keyword search misses articles titled 'terminate subscription' or 'close my plan'. Semantic search finds them because meaning matches, not words.

◉ Grounded AI Outputs: Instead of an LLM inventing answers from training data, RAG retrieves real documents first. The model reads them, then responds. Hallucinations drop dramatically.

◇ Personalisation at Scale: Embeddings let you represent users and content in the same space. Find content close to a user's interest vector. No hand-crafted rules — the geometry does the work.

02 · 1. Embeddings Intuition

Embeddings Intuition & Semantic Search

Establish a solid mathematical and intuitive understanding of high-dimensional semantic spaces.

Choose Your Model OpenAI Embeddings API

text-embedding-3-small

  • 1536 dimensions.

  • $0.02 per million tokens.

  • Best choice for most production RAG systems — excellent quality-to-cost ratio.

  • Use for: FAQ search, support Q&A, docs retrieval.

  • This is what today's lab uses.

text-embedding-3-large

  • 3072 dimensions.

  • $0.13 per million tokens — 6.5× more expensive.

  • Use for: code search, multilingual retrieval, fine-grained ranking where small quality differences matter.

  • Doubles vector storage size.

Embeddings Without Math

  • Think of embeddings as GPS coordinates for meaning — 'bank' (financial) and 'bank' (river) get different coordinates because their context differs

  • You don't need to understand the math to use embeddings — you need to understand what they encode and what they cannot encode

  • Embeddings are trained on billions of text examples — the model learns that 'cancel subscription' and 'terminate my account' appear in similar contexts

  • The output is always a fixed-length vector regardless of input length — one sentence and one paragraph both become 1536 numbers

  • Distance in embedding space = semantic distance. Close vectors = similar meaning. Far vectors = different meaning.

Semantic Similarity

  • Two pieces of text are semantically similar if they convey the same information, even with completely different words

  • 'How do I cancel?' and 'What is the termination process?' are semantically similar — keyword search misses this, embeddings catch it

  • Similarity is symmetric: if A is close to B, then B is close to A — embeddings respect this geometry

  • Semantic similarity is not the same as factual correctness — two wrong answers about the same topic score as similar

  • This has a practical implication: embeddings retrieve text that sounds relevant, not text that is accurate — your grounding step must validate

Vector Intuition

  • A vector is just a list of numbers: [0.21, -0.84, 0.13, ...] with 1536 entries for text-embedding-3-small

  • Each dimension loosely encodes some semantic feature — though no single dimension has a clean human-readable interpretation

  • The direction of a vector matters more than its magnitude — cosine similarity measures angle, not distance

  • Adding and subtracting vectors can encode relationships: king - man + woman ≈ queen (the famous Word2Vec result)

  • You don't store or inspect individual dimensions — you compare vectors using similarity functions, and the database does the nearest-neighbour search

Why Embeddings Power AI Search

  • Traditional search: tokenise query → match exact or stemmed tokens → rank by TF-IDF or BM25. Breaks on vocabulary mismatch.

  • Embedding search: encode query → find nearest vectors → return semantically similar documents. Vocabulary-independent.

  • Embeddings also enable cross-modal search — same space can encode text and images, enabling image search from text queries

  • Every major AI product (ChatGPT memory, Notion AI, GitHub Copilot, Perplexity) uses embedding retrieval at the core

  • The engineering decision is not whether to use embeddings — it's which model, which chunk size, and which vector store

Similarity Scoring

  • 1.0 — Perfect match: identical or near-duplicate text

  • 0.85+ — Strong semantic match: different wording, same intent

  • 0.60–0.85 — Related topic: thematically connected but different focus

  • <0.50 — Weak or unrelated: likely retrieval noise, should be filtered

Comparative Analysis Cosine Similarity Concept

The Formula

  • cosine_similarity(A, B) = dot(A, B) / (|A| × |B|).

  • Ranges from -1 (opposite) to +1 (identical direction).

  • Ignores vector magnitude — only the direction matters.

  • This makes it robust to text length differences.

Why It Works for Text

  • A long document and a short sentence about the same topic point in the same direction even if their magnitudes differ.

  • Euclidean distance would penalise the long document unfairly.

  • Cosine similarity is length-normalised by design.

Comparative Analysis Embeddings vs Keywords

Keywords Win When

  • Exact phrase must match (product SKU, error code, legal citation).

  • Query and document use identical terminology.

  • You need explainability — keyword match is auditable.

  • Latency is critical — BM25 is faster than ANN search at small scale.

Embeddings Win When

  • Users paraphrase naturally.

  • Synonyms and related concepts must match.

  • Cross-lingual retrieval is needed.

  • Documents are long and varied.

  • The query is a natural-language question, not a search keyword.

  • Best practice: hybrid search uses both.

Personalization via Embeddings

  • Represent users as vectors by averaging the embeddings of their viewed/liked content — this is a 'user interest vector'

  • At query time, bias retrieval toward content close to the user vector — personalised without hand-coded rules

  • Netflix, Spotify, LinkedIn, and Amazon all use embedding-based personalisation at their core

  • Cold start problem: new users have no history → no user vector → use demographic or onboarding signals as a proxy

  • Privacy implication: user interest vectors can be highly sensitive — treat them as PII, encrypt at rest, scope access tightly

Semantic Clustering Intuition

  • Cluster embeddings with k-means or HDBSCAN to discover topic groups in your document corpus — no labels needed

  • Support tickets cluster into: billing issues, login problems, feature requests, performance complaints — automatically

  • Cluster centroids become 'topic anchors' — useful for routing queries to specialised retrieval indexes

  • Outliers in embedding space signal novel queries or content that doesn't fit existing patterns — worth flagging for review

  • This is how LLM providers detect prompt injection clusters and misuse patterns across millions of API calls

03 · 2. RAG Fundamentals

RAG Fundamentals & Orchestration

Master the core Retrieve-Augment-Generate orchestration flow and understand structural tool design frameworks.

Retrieve → Augment → Generate

  1. Step 1 — Retrieve: Embed the user query. Run nearest-neighbour search against your vector index. Return top-k most semantically similar document chunks.

  2. Step 2 — Augment: Insert retrieved chunks into the LLM prompt as context. The prompt now contains the question AND the relevant knowledge. The LLM is grounded, not guessing.

  3. Step 3 — Generate: LLM reads the retrieved context and generates an answer. It should cite the source chunks. If the answer isn't in the context, it should say so — not hallucinate.

Why RAG Exists

  • LLMs have a knowledge cutoff — they don't know about events, documents, or data created after training

  • LLMs hallucinate confidently — they generate plausible-sounding text even when they don't have the facts

  • Fine-tuning is expensive, slow, and requires retraining every time your data changes — not practical for live knowledge bases

  • RAG lets the model access live, proprietary, or domain-specific knowledge without any model weight changes

  • Every enterprise AI product that needs to answer questions from internal docs uses RAG — it is now a baseline engineering skill

Comparative Analysis RAG vs Fine-Tuning

RAG

  • Knowledge is external — in a vector index.

  • Update the index, not the model.

  • Works with proprietary data without exposing it during training.

  • Lower cost, faster iteration.

  • Best for: factual Q&A over a changing knowledge base, support chatbots, document search.

Fine-Tuning

  • Knowledge is baked into model weights.

  • Expensive GPU time per training run.

  • Data used in training may raise privacy concerns.

  • Best for: style transfer, domain-specific language patterns, consistent output format across thousands of calls.

  • Not a substitute for RAG on factual tasks.

Comparative Analysis RAG vs Pure Prompting

Pure Prompting

  • Paste all relevant context manually into the system prompt.

  • Works for small, static knowledge bases.

  • Breaks at scale — context window has limits.

  • Every call sends the full context, raising cost.

  • No dynamic retrieval — you decide what's relevant upfront.

RAG

  • Retrieve only what is relevant to this specific query.

  • Scales to millions of documents — only top-k chunks enter the context window.

  • Cost-efficient — shorter prompts per call.

  • Context is dynamic and query-driven.

  • Requires engineering: embedding pipeline, vector index, retrieval evaluation.

Hallucination Reduction

  • Hallucination = LLM generates confident text that is factually wrong, because it has no source to ground the answer

  • RAG grounds every answer in retrieved text — the model reads before it generates, not just predicts from training weights

  • Add an instruction: 'If the answer is not in the provided context, say you don't know' — this is grounding enforcement

  • Hallucination still occurs in RAG if: retrieved chunks are wrong, irrelevant chunks are retrieved, or the model ignores the context

  • Measure grounding rate: for each response, check whether every factual claim traces to a retrieved chunk. Automate this with an LLM-as-judge eval.

Retrieval Quality Problems

  • Wrong retrieval is worse than no retrieval — a confidently wrong context chunk misleads the LLM into a confident wrong answer

  • Retrieval failures: semantic drift (vague queries match unrelated chunks), chunk fragmentation (answer split across chunks), vocabulary gaps (query uses terms not in the corpus)

  • Ranking failures: top-k returns technically similar but contextually irrelevant chunks — similarity score alone is a poor quality signal

  • Retrieval evaluation is non-trivial: you need labelled query→document pairs to measure recall@k and precision@k

  • Start simple: log queries and retrieved chunks in production, manually review 20 per week — this catches 80% of retrieval failures

Grounding Outputs

  • Grounding = ensuring every LLM claim can be traced to a retrieved source document

  • Implementation: include source metadata (document title, chunk ID, URL) alongside each retrieved chunk in the prompt

  • Instruct the model to cite sources inline: 'According to [Help Article #42]: ...' — users can verify claims

  • Grounding also enables auditability in regulated industries: financial advice, medical Q&A, legal search all require traceable outputs

  • Anti-pattern: letting the LLM synthesise across multiple chunks without attribution — the source of each claim becomes unclear

Multi-Step RAG

  1. Step 1 — Query Decomposition: Complex question: 'How does our cancellation policy differ for monthly vs annual plans?' → Decompose into 2 sub-queries: cancellation policy for monthly plans, cancellation policy for annual plans.

  2. Step 2 — Parallel Retrieval: Run both sub-queries against the vector index simultaneously. Retrieve top-k chunks for each. This ensures both aspects of the question are covered.

  3. Step 3 — Re-ranking: Merge retrieved chunks. Re-rank by relevance to the original complex question. Remove duplicates and low-quality chunks.

  4. Step 4 — Synthesise: LLM reads merged, re-ranked context. Generates a comparative answer that addresses both sub-questions. Cites sources for each claim.

Retrieval Orchestration

  • Orchestration = deciding what to retrieve, when, and how to combine results before passing to the LLM

  • LangChain provides RetrievalQA chain: wraps retriever + LLM + prompt template into one callable object

  • LlamaIndex provides SubQuestionQueryEngine: automatically decomposes complex questions and routes sub-queries

  • Router retrievers: detect query type (e.g. billing vs. technical vs. policy) and dispatch to specialised indexes

  • Agentic RAG: LLM decides whether to retrieve and what to search for — retrieval becomes a tool call, not a fixed pipeline step

Choose the Right Tool LangChain vs LlamaIndex

LangChain

  • Broad orchestration framework.

  • Best for: custom RAG pipelines, chaining multiple steps, mixing retrievers with agents and tools.

  • More code, more control.

  • Use when you need flexibility across the full AI stack beyond just retrieval.

LlamaIndex

  • Purpose-built for document ingestion and retrieval.

  • Best for: indexing PDFs, tables, structured data, large document sets quickly.

  • Less boilerplate for pure RAG.

  • Use when your task is: load documents → index → query — and you do not need custom pipeline logic.

04 · 3. Chunking & Retrieval

Chunking & Retrieval Engineering

Engineer optimal chunking strategies, semantic partitioning, and robust relevance indexing.

Why Chunking Matters

  • LLMs have context window limits — you cannot embed an entire 200-page document and retrieve it whole

  • Chunking splits documents into smaller units that are independently embedded and stored as separate vectors

  • Each chunk gets its own embedding — meaning its similarity score is computed independently against any query

  • The chunk is the unit of retrieval: if the answer is split across two chunks, retrieval may fail even if both chunks are in the index

  • Chunking strategy directly determines retrieval quality — it is one of the highest-leverage engineering decisions in a RAG system

Comparative Analysis Chunk Size Tradeoffs

Small Chunks (128–256 tokens)

  • Pros: precise retrieval, low noise in context window, easy to attribute to a specific source sentence.

  • Cons: answers that span multiple sentences get fragmented — retrieval returns incomplete information.

  • Better for: precise fact lookup, Q&A with short factual answers.

Large Chunks (512–1024 tokens)

  • Pros: preserves full context around an answer, reduces fragmentation risk.

  • Cons: retrieved chunks contain more irrelevant content, increasing LLM confusion and prompt cost.

  • Better for: narrative explanation, complex multi-part answers, technical documentation with long reasoning chains.

Note: 256–512 tokens is the typical production sweet spot for RAG; 50–100 token overlap is standard. [FIXED: added sweet-spot callout to match current best practice of 256–512 token chunks]

Overlap Strategies

  • Overlap = repeating N tokens at the end of one chunk at the start of the next, so boundary-spanning answers appear in both chunks

  • RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50): 50 tokens of overlap between adjacent chunks

  • Overlap increases total index size proportionally — 10% overlap → ~10% more vectors → ~10% more storage and embedding cost

  • Too little overlap: answers that straddle a chunk boundary are split and neither chunk retrieves the full answer

  • Too much overlap: redundant content floods the context window; re-ranking becomes harder because near-duplicate chunks score similarly

Metadata Design

  • Every chunk should carry metadata: source document, section title, page number, last updated timestamp, author

  • Metadata enables filtered retrieval: 'find similar chunks, but only from the billing category' — reduces noise dramatically

  • LangChain Document objects carry a metadata dict alongside page_content — store whatever you'll need to filter on

  • Source metadata enables citation: when the LLM answers, it can cite 'Help Article: Cancellation Policy, Section 3' rather than an opaque chunk ID

  • Timestamp metadata enables freshness filtering: exclude chunks from documents last updated >6 months ago for time-sensitive queries

Similarity Search

  • Exact nearest neighbour: compare query vector to every vector in the index, return the closest k. O(n) per query — too slow for large indexes.

  • Approximate Nearest Neighbour (ANN): HNSW, IVF, and ANNOY trade tiny accuracy loss for massive speed gains — O(log n) or sublinear

  • FAISS (Facebook AI Similarity Search): fastest open-source ANN library, runs locally, used in production at Meta scale

  • pgvector: adds vector similarity search to Postgres using IVF or HNSW indexes — query syntax: ORDER BY embedding <-> query_embedding LIMIT k

  • Similarity threshold: FAISS returns L2 distance (lower = more similar). pgvector <-> operator = L2 distance, <=> = cosine distance [FIXED: corrected operator mapping — pgvector <-> is L2, <=> is cosine; original had the labels correct but worth making explicit]

Retrieval Ranking

  • Pure similarity ranking has a problem: the most similar chunk may not be the most useful for the query

  • Re-ranking: use a cross-encoder model to score each retrieved chunk against the query — more accurate but slower than bi-encoder similarity

  • Cross-encoders process query+chunk together — they capture interaction between query and document that bi-encoder similarity misses

  • LangChain CohereRerank and FlashrankRerank provide drop-in re-ranking after initial retrieval [FIXED: Cohere Rerank API is the correct product name; ColBERT and cross-encoder models are also standard rerankers]

  • Practical approach: retrieve top-20 by similarity, re-rank, pass top-3 to the LLM — balances speed and accuracy

Top-k Retrieval

  • k = number of chunks returned by the retriever and passed to the LLM as context

  • Too small k (k=1): high precision, high miss rate — if the top result is wrong, there's no fallback

  • Too large k (k=20): high recall, but context window fills with noise — LLM has more to ignore, cost rises, quality drops

  • k=3–5 is the starting point for most RAG systems; measure and tune against your specific query distribution

  • Adaptive k: retrieve more chunks for complex queries, fewer for simple factual lookups — LangChain supports this via custom retriever logic

Comparative Analysis Hybrid Search

Semantic Search (Embedding)

  • Finds semantically similar content even with vocabulary mismatch.

  • Handles paraphrases, synonyms, related concepts.

  • Fails on: exact product codes, version numbers, error codes, named entities with unusual spelling.

Hybrid Search

  • Run both BM25 (keyword) and embedding search.

  • Merge results with Reciprocal Rank Fusion (RRF).

  • Each result is re-ranked by its combined score.

  • Best of both worlds: semantic coverage + exact term precision.

  • LangChain EnsembleRetriever supports this natively.

Retrieval Failure Modes

  • Semantic drift: vague or ambiguous queries match topically unrelated chunks — 'what's your policy?' matches too many documents

  • Vocabulary gap: user uses jargon the corpus doesn't contain — embeddings encode from training data, not your specific glossary

  • Chunk fragmentation: answer spans two adjacent chunks, neither alone is sufficient — overlap reduces but doesn't eliminate this

  • Stale index: document updated in the source, but embedding index not rebuilt — retrieved chunks are outdated

  • Adversarial queries: carefully crafted queries that exploit the embedding space to retrieve unintended content — a security concern for public-facing RAG

Semantic Drift in Retrieval

  • Semantic drift: the retrieved documents shift topic relative to the original user intent, especially with short or vague queries

  • Example: query 'tell me about plans' retrieves pricing plans, project plans, and exercise plans — all semantically adjacent to 'plans'

  • Mitigation 1: query expansion — use the LLM to rewrite the query into a more specific retrieval query before embedding

  • Mitigation 2: metadata filtering — restrict retrieval to a relevant document category before running similarity search

  • Mitigation 3: intent classification — classify query intent first, then route to a specialised index for that intent

Comparative Analysis Over-Retrieval vs Under-Retrieval

Over-Retrieval (too many chunks)

  • Context window fills with weakly related content.

  • LLM gets confused — may ignore the actually relevant chunk buried in noise.

  • Prompt cost rises.

  • Quality drops.

  • Fix: lower k, add similarity threshold filter, re-rank aggressively.

Under-Retrieval (too few chunks)

  • Answer not present in retrieved context — LLM either says it doesn't know (good) or hallucinates (bad).

  • Miss rate rises.

  • Fix: increase k, use hybrid search, check chunk size (may be too large or fragmented).

05 · 4. Vector Databases

Vector Databases & pgvector Integration

Deploy Postgres-native vector stores alongside self-managed and hosted databases.

pgvector Overview

  • pgvector is a Postgres extension that adds a native vector data type and similarity search operators to standard SQL

  • Install: CREATE EXTENSION IF NOT EXISTS vector; — available on Supabase, Neon, RDS, and most managed Postgres providers

  • Store vectors: ALTER TABLE items ADD COLUMN embedding vector(1536); — dimensionality is a schema constraint

  • Query: SELECT * FROM items ORDER BY embedding <-> query_embedding LIMIT 5; — <-> is L2 distance, <=> is cosine distance

  • Index for performance: CREATE INDEX ON items USING hnsw (embedding vector_cosine_ops); — HNSW gives ANN performance inside Postgres

Why pgvector Is Product-Native

  • If your product already runs on Postgres, pgvector adds vector search without introducing a second database system

  • Single database = single transaction: INSERT a product row and its embedding in one ACID transaction — no sync job, no eventual consistency

  • Join vectors with relational data: SELECT * FROM products JOIN embeddings ON products.id = embeddings.product_id WHERE products.category = 'billing' ORDER BY embedding <-> $1 LIMIT 5

  • Operational simplicity: one backup strategy, one monitoring system, one access control model, one connection pool

  • Supabase ships pgvector by default — enable with one SQL command, query from any Postgres client

ACID Transactions & Vector Consistency

  • ACID = Atomicity, Consistency, Isolation, Durability — the guarantees that make databases reliable for production systems

  • With pgvector: INSERT INTO products (name, embedding) VALUES ($1, $2) — both the product row and its embedding commit or both roll back. Atomic by default.

  • Without ACID across two systems: write to Postgres succeeds, write to Pinecone fails — now your relational data and vector index are out of sync

  • Consistency guarantee: every search result in pgvector reflects the current state of the database — no stale reads from an async sync job

  • This is not a theoretical concern: dual-write sync failures are a common production incident in systems that split vectors into a separate store

Comparative Analysis The Dual-Write Sync Problem

Pinecone + Postgres Pattern

  • Write product data to Postgres.

  • Then write embedding to Pinecone.

  • If step 2 fails: product exists in Postgres, no vector in Pinecone — it's invisible to semantic search.

  • Fix requires: a sync job, retry logic, a reconciliation queue, and monitoring for drift.

  • This is a distributed systems problem you now own.

pgvector Pattern

  • Write product data AND embedding to Postgres in one transaction.

  • If the transaction fails: both roll back — database stays consistent.

  • No sync job.

  • No reconciliation.

  • No distributed state.

  • The embedding lives where the data lives.

  • Operational complexity stays proportional to scale.

Why Pinecone + Postgres Creates Distributed State

  • Two databases = two sources of truth = a distributed system with all its coordination problems

  • Every write now requires agreement between two systems — network partition, API failure, or timeout breaks the contract

  • Delete a product from Postgres: did you delete the vector from Pinecone? Out-of-sync deletes cause ghost results in search — items returned by semantic search that no longer exist

  • Schema changes: add a column to Postgres, update the embedding model — now you need to coordinate migration across two systems simultaneously

  • At small scale this is manageable. At 50,000 users with millions of records, the sync overhead becomes an engineering team tax.

Why pgvector Simplifies Architecture

  • Single system of record: Postgres is the authoritative store for both relational data and vectors — no reconciliation needed

  • SQL joins between vectors and metadata: filter by category, date, user_id in the same query as vector similarity search

  • Familiar tooling: existing Postgres backups, monitoring, migrations, and ORM support all work unchanged

  • pgvector HNSW index brings performance within range of dedicated vector databases for most SaaS-scale workloads (<10M vectors)

  • When does pgvector not scale? Very high query throughput (>10k QPS) or very large indexes (>100M vectors) — at that scale, dedicated vector infrastructure is warranted

FAISS Overview

  • FAISS (Facebook AI Similarity Search) is an open-source library for efficient similarity search and clustering of dense vectors

  • Runs entirely in-process — no server, no network, no API. Index lives in memory (or saved to disk as a file).

  • faiss-cpu installs as a Python package: from langchain_community.vectorstores import FAISS — LangChain wraps FAISS natively

  • Save and load: vectorstore.save_local('my_index') · FAISS.load_local('my_index', embeddings) — persist across runs

  • Best for: local development, offline indexes, prototyping, CI/CD retrieval testing. Not for production multi-user systems — no concurrent write support.

Comparative Analysis Hosted vs Self-Managed Vectors

Hosted (Pinecone, Weaviate Cloud)

  • No infrastructure ops.

  • Auto-scaling.

  • Managed ANN indexing.

  • Higher per-query cost at scale.

  • Vendor lock-in risk.

  • Adds a second API dependency.

  • Best when: team lacks Postgres/ops expertise, need >10M vectors from day one, or product needs multi-region vector search.

Self-Managed (pgvector, FAISS)

  • Full control.

  • No per-query pricing.

  • Integrates with existing Postgres stack.

  • Requires your own ops for backup, scaling, monitoring.

  • pgvector handles most SaaS workloads.

  • FAISS for local/offline.

  • Best when: already on Postgres, team has ops capacity, or data residency requirements.

Indexing Pipelines

  1. Ingest — Load raw documents: PDFs, markdown files, CSVs, database rows, API responses. LlamaIndex SimpleDirectoryReader or LangChain DocumentLoaders handle most formats.

  2. Chunk — Split into retrieval units: RecursiveCharacterTextSplitter with tuned chunk_size and chunk_overlap. Attach metadata: source, section, timestamp.

  3. Embed — Generate vectors: OpenAI Embeddings API: one API call per chunk. Batch in groups of 100+ for cost efficiency. Cache embeddings to avoid re-embedding unchanged content.

  4. Index — Store in vector database: FAISS.from_documents() for local. pgvector INSERT for Supabase. Pinecone upsert for hosted. Build HNSW index after bulk load for query performance.

Bridging the Model to the Database

  • An embedding model is not interchangeable with any other — switching models requires rebuilding the entire index

  • text-embedding-3-small produces 1536-dimensional vectors. text-embedding-3-large produces 3072. These are incompatible — you cannot mix them in one index.

  • The dimensionality is a schema contract: CREATE TABLE items (embedding vector(1536)) — changing this requires ALTER TABLE and full re-indexing

  • Every query must be embedded with the same model as the index — different model = different vector space = meaningless similarity scores

  • Document this contract in your project: MODEL=text-embedding-3-small, DIMENSIONS=1536, INDEX=faq_hnsw. Treat a model change like a database migration.

Embedding Dimensionality as Schema Contract

  • 1536 — text-embedding-3-small: default choice for most RAG systems

  • 3072 — text-embedding-3-large: use only when 1536 is measured to be insufficient

  • 0 — Tolerance for dimension mismatch: pgvector and FAISS both reject mismatched dimensions immediately

  • 100% — Index rebuild required on model change: no partial migration possible

Comparative Analysis 1536 vs 3072 Dimensions

1536 Dimensions (small)

  • 6KB per vector at float32. [FIXED: 1536 dims × 4 bytes = 6,144 bytes ≈ 6KB, not 64KB; original said 64KB which is incorrect]

  • 1M vectors = ~6GB RAM for FAISS in-memory index.

  • Supabase free tier handles ~100K vectors comfortably.

  • Cosine search latency: <10ms at 1M vectors with HNSW.

  • The right default for 99% of SaaS RAG systems.

3072 Dimensions (large)

  • 12KB per vector at float32. [FIXED: 3072 dims × 4 bytes = 12,288 bytes ≈ 12KB, not 128KB; original said 128KB which is incorrect]

  • 1M vectors = ~12GB RAM — doubles storage and memory.

  • Marginally better retrieval quality for code, multilingual, and fine-grained ranking.

  • 6.5× more expensive to generate.

  • Only worth it when you've measured that 1536 isn't good enough.

Why Mismatched Dimensions Fail Instantly

  • pgvector enforces dimensionality at the column level: vector(1536) rejects any INSERT with a vector of different length

  • FAISS indexes are built with a fixed dimension — querying with a different-dimension vector throws an immediate exception

  • This is by design: similarity search requires vectors in the same geometric space — different dimensions = different spaces, comparison is meaningless

  • The failure is loud and immediate in development, which is good. The danger is subtle: indexing with model A, querying with model B — same dimension, different space, silently wrong results.

  • Test your embedding pipeline with dimension assertions: assert len(embedding) == 1536, f'Expected 1536, got {len(embedding)}'

Model Migration Implications

  • Switching embedding models (e.g. text-embedding-ada-002 → text-embedding-3-small) requires full re-embedding of every document in your corpus

  • You cannot partially migrate: mixing embeddings from two models in one index produces meaningless similarity scores across the boundary

  • Migration strategy: build the new index in parallel, validate retrieval quality on test queries, then cut over — never in-place

  • Cost estimate: 10M tokens of corpus × $0.02/1M = $0.20 to re-embed with text-embedding-3-small. Usually trivial.

  • Operational risk: if your production index and query pipeline are updated at different times, you have a split-brain embedding problem — queries return garbage until both sides are on the same model

Index Rebuild Requirements

  • Index rebuild is required when: switching embedding models, changing vector dimensions, bulk-updating document content, correcting a systematic chunking error

  • HNSW index rebuild in pgvector: DROP INDEX idx_name; CREATE INDEX ON table USING hnsw (embedding vector_cosine_ops); — takes minutes for <1M vectors

  • FAISS index rebuild: delete saved index files, re-run FAISS.from_documents() with new embeddings — straightforward but requires re-embedding all chunks

  • Zero-downtime rebuild: write new index to a shadow table/collection, validate quality, then swap — keep old index live until new one is verified

  • Automate rebuild triggers: CI pipeline detects model version change in config → triggers re-indexing job → notifies team on completion

06 · 5. App Architecture

Semantic Search Application Architecture

Construct a complete end-to-end FastAPI FAQ search endpoint wired into a debounced React UI client.

Embedding Generation Pipeline

  1. Input — Raw text arrives: User query, document chunk, product description, support ticket — any text that needs a vector representation.

  2. Preprocess — Clean and normalise: Strip HTML, normalize whitespace, truncate to model token limit (8191 tokens for text-embedding-3-small). Log token counts for cost tracking.

  3. Embed — Call OpenAI Embeddings API: client.embeddings.create(model='text-embedding-3-small', input=text). Returns embedding list. Cache result keyed by hash(text) to avoid re-embedding identical content.

  4. Store — Persist to vector index: INSERT into pgvector table or FAISS index. Include metadata. Confirm write before returning success to caller.

Query Embedding Flow

  • At query time: user types a question → embed the question with the same model used to index documents → retrieve nearest neighbours

  • Query embedding is a real-time operation — it must complete in <100ms to meet typical UI latency budgets

  • text-embedding-3-small embedding latency: ~50ms per call including network — acceptable for synchronous search endpoints

  • Never cache query embeddings by query string — users rarely type identical queries and caching adds complexity for minimal gain

  • Query preprocessing: lowercase, strip special characters, handle empty string edge case before sending to embedding API

Retrieval Lifecycle

  1. Query arrives — Validate and embed: Validate input length. Embed with text-embedding-3-small. Log query + embedding hash for observability.

  2. Search — ANN lookup in vector index: pgvector HNSW or FAISS IVF returns top-k candidates with similarity scores. Apply metadata filters if needed (category, date range).

  3. Re-rank — Optional cross-encoder scoring: If using re-ranking: score top-20 candidates with cross-encoder, return top-3 after re-ranking. Adds ~100ms latency.

  4. Augment — Build LLM context: Format retrieved chunks with source metadata into prompt context. Enforce context length limit — truncate if needed.

  5. Generate — LLM produces grounded response: LLM reads context and generates answer with citations. Log: retrieved chunks, similarity scores, prompt, response, latency.

React Semantic Search UX

  • Search input → debounced hook (300ms) → API call on stable value → display results with scores

  • Show similarity as a percentage: Math.round((1 - score) * 100) + '% match' — FAISS L2 score needs conversion, pgvector cosine distance maps directly via (1 - distance) [FIXED: clarified that pgvector cosine distance converts to similarity via 1 - distance, not "maps directly"]

  • Display top-3 results as cards: content snippet, source label, relevance percentage — not a ranked list, a retrieval explanation

  • Loading state: show skeleton cards during embedding + retrieval (~200ms total) — never block the UI on a pending search

  • Empty state: 'No results found' with a suggestion to rephrase — don't hide the failure, help the user adjust their query

Debounced Input

  • Without debounce: every keystroke triggers an embedding API call — 5 keystrokes = 5 API calls, 5 OpenAI charges, 5 in-flight requests

  • useDebounce hook: delay the search trigger until the user stops typing for N milliseconds — typically 300ms for search

  • Implementation: const debouncedQuery = useDebounce(query, 300) · useEffect(() => { if (debouncedQuery) fetchResults(debouncedQuery) }, [debouncedQuery])

  • 300ms feels instant to users but eliminates ~80% of API calls compared to per-keystroke search

  • Cancel in-flight requests: use AbortController to cancel the previous fetch when a new debounced query fires — prevents stale results appearing after a fast typer

Comparative Analysis Re-ranking Concepts

Bi-encoder (Embedding Similarity)

  • Embed query once.

  • Embed each document once at index time.

  • Compare vectors at query time.

  • Fast — O(log n) with ANN index.

  • But: query and document are encoded independently, so cross-term interaction is lost.

Cross-encoder (Re-ranking)

  • Encode query + candidate document together.

  • Model sees both simultaneously — captures interaction.

  • Much more accurate ranking.

  • Slow — O(k) inference per query, where k is number of candidates.

  • Pattern: retrieve top-20 with bi-encoder, re-rank with cross-encoder, pass top-3 to LLM.

Similarity Score Interpretation

  • pgvector cosine distance: 0 = identical, 2 = opposite. Convert to similarity: 1 - cosine_distance. 0.9+ = strong match.

  • FAISS L2 distance: 0 = identical, increases with distance. Lower is better. No natural upper bound — scale depends on vector norms.

  • Display to users: convert to a 0–100% relevance scale and show it — users understand 'this result is 92% relevant'

  • Use scores as a filter, not just a rank: discard results below a threshold (e.g. similarity < 0.7) rather than always showing top-k regardless of quality

  • Never trust score alone for quality: a 0.95 similarity to a well-written but wrong chunk is more dangerous than a 0.75 to the correct chunk

Search Quality Evaluation

  • Recall@k: of all relevant documents for a query, what fraction appear in the top-k results? Measures coverage.

  • Precision@k: of the k returned results, what fraction are actually relevant? Measures noise.

  • MRR (Mean Reciprocal Rank): average of 1/rank of the first relevant result — measures whether the best result appears near the top

  • Build an evaluation set: 50–100 test query → expected document pairs. Run retrieval pipeline. Compute recall@3 and precision@3.

  • RAGAS framework: automated RAG evaluation using LLM-as-judge for faithfulness, relevance, and context precision — run in CI

Retrieval Observability

  • Log every retrieval event: query, retrieved chunk IDs, similarity scores, latency, whether the user found the result useful

  • Track query miss rate: queries where top-1 similarity score falls below your threshold — these are retrieval failures

  • Monitor embedding API latency and error rate separately from retrieval latency — they fail in different ways

  • Weekly manual review: sample 20 query+result pairs from logs. Identify systematic failures. Adjust chunking or retrieval strategy.

  • LangSmith (LangChain's observability platform): traces every retrieval step, logs prompts, measures token usage — free tier available

Grounded Response Generation

  • Prompt template: 'Answer the question using ONLY the context below. If the answer is not in the context, say you don't know. Context: {retrieved_chunks}. Question: {query}'

  • Include source metadata in context: 'Source: Help Article #42, Section: Cancellation Policy' — the LLM will include it in citations

  • Temperature = 0 for grounded Q&A: deterministic responses reduce hallucination risk. No need for creativity when the answer is in the context.

  • Post-generation validation: use an LLM-as-judge to check whether every claim in the response is supported by a retrieved chunk

  • Streaming responses: use OpenAI streaming API to show the response as it generates — users perceive lower latency even if total time is the same

// App.tsx
import React, { useState, useEffect } from 'react';

// Custom debounce hook to minimize API rate-limit hits
function useDebounce<T>(value: T, delay: number): T {
  const [debouncedValue, setDebouncedValue] = useState<T>(value);

  useEffect(() => {
    const handler = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(handler);
  }, [value, delay]);

  return debouncedValue;
}

  content: string;
  score: number;
}

Enterprise Product Use Cases

Analyze real-world architectures, compliance limits, and permissioned internal search frameworks.

Support Q&A

  • Index your help articles, policy documents, and product documentation into a vector store

  • User asks a natural-language question → retrieve most relevant articles → LLM synthesises a grounded answer with citations

  • Deflection rate: measure what % of support tickets are fully resolved by the RAG system without human escalation — typical target: 40–60%

  • Confidence gating: if top retrieved chunk similarity is below threshold, escalate to human agent rather than risk a low-quality answer

  • Key metric: customer satisfaction score for RAG-answered tickets vs human-answered — validate that deflection doesn't hurt CSAT

Docs Search

  • Developers search for: how to use a specific API, what a config option does, why an error occurs — natural language, not keyword lookup

  • Index all SDK docs, API references, changelog, and tutorials as separate document types with metadata

  • Query routing: detect if the query is about an API endpoint (route to API reference index) vs a conceptual question (route to tutorials)

  • Version awareness: tag chunks with version metadata — search results can be filtered to the version the user is on

  • GitHub Copilot's docs Q&A, Stripe's AI search, and Vercel's AI assistant all use this pattern

Knowledge Assistants

  • Internal knowledge base: index Confluence, Notion, Slack threads, internal wikis, meeting transcripts, runbooks

  • Query: 'What's our policy on expense reimbursement?' retrieves the HR policy chunk and generates a grounded answer

  • Freshness critical: internal knowledge changes fast — rebuild index on document update, not on a weekly schedule

  • Access control: different employees should only retrieve documents they have permission to access — metadata-based ACL filtering at query time

  • High-value starting point: index your engineering runbooks and onboarding docs — new engineers get instant, cited answers to setup questions

Personalization

  • Build a user interest vector: average embeddings of articles/products the user engaged with — this is their semantic profile

  • At recommendation time: retrieve content whose embeddings are closest to the user vector — no explicit category tags needed

  • Combine with collaborative filtering: weight the user vector toward content similar users also engaged with

  • Cold start: new users have no history → use embedding of their onboarding responses or stated interests as a starting user vector

  • Privacy: user vectors are sensitive — they encode preferences and interests in computable form. Encrypt, scope access, and provide opt-out.

Recommendation Systems

  • 'More like this': embed the current item, retrieve nearest neighbours from the product/content catalog — instant recommendations

  • Cross-domain recommendations: embed both products and blog articles in the same space — recommend articles related to what a user bought

  • Real-time personalisation: recompute user vector on each session by averaging the last N interactions — no daily batch job needed

  • Diversity: nearest-neighbour retrieval can create filter bubbles — add diversity penalty to push recommendations slightly outside the cluster centroid

  • A/B test: compare embedding-based recommendations vs rule-based 'customers also bought' — measure click-through rate and conversion

Enterprise Search

  • Enterprise data lives in silos: Salesforce, Jira, Confluence, email, Slack, Google Drive — keyword search across all fails

  • Unified embedding index: crawl all sources, chunk, embed with consistent model, store in one vector index with source metadata

  • Permission-aware retrieval: filter vectors at query time by document ACL — return only content the querying user can access

  • Query understanding: 'Q3 revenue forecast from the finance team' → needs entity detection (Q3, finance team) + semantic retrieval

  • Production challenge: keeping the index fresh when source data changes across 10 different systems — event-driven ingestion > polling

Privacy Considerations

  • PII in the corpus: if your index contains customer names, emails, or medical records, retrieval may expose this data to other users — scope access with ACL filtering

  • Embedding model data retention: OpenAI does not train on API inputs by default, but check your data processing agreement for your compliance jurisdiction

  • User vectors as PII: a user's interest embedding encodes their preferences in computable form — treat it as PII under GDPR and CCPA

  • Data residency: some regulated industries (healthcare, finance, EU GDPR) require data to stay in specific regions — check your vector store's data residency options

  • Audit trail: log what was retrieved in response to what query by which user — required for compliance in regulated RAG deployments

Week 1 Review & Consolidation

Week 1 Recap

◈ Foundation: AI capabilities and limits. Token economics. Context window management. Model selection. Prompt anatomy. You know what LLMs can and cannot do — and why.

◉ Prompting: Zero-shot, few-shot, chain-of-thought. System prompts. Structured output. Evaluation. You can reliably extract structured data and control LLM behaviour through prompt design.

◇ Integration: OpenAI API, streaming, function calling. RAG pipelines. Embeddings and semantic search. Vector databases. You can wire AI into a real product, not just a demo.

Mental Model Review

  • LLMs are next-token predictors trained on human-generated text — they complete patterns, they do not reason from first principles

  • The context window is the model's working memory — everything the model knows during a generation comes from what's in the window

  • Temperature controls creativity: 0 = deterministic, 1 = creative, >1 = incoherent. For factual tasks, use 0.

  • Token cost is not just price — longer contexts mean slower responses. Compress context before scaling.

  • LLMs have no persistent memory between API calls — every call starts fresh unless you explicitly include conversation history in the prompt

Prompting Review

  • Zero-shot: just describe the task clearly. Works for simple classification, summarisation, formatting, and translation.

  • Few-shot: provide 2–5 examples of input→output pairs before the task. Dramatically improves structured output accuracy.

  • Chain-of-thought: 'Think step by step' or explicit reasoning steps. Improves accuracy on multi-step reasoning tasks.

  • System prompt separation: put instructions, constraints, and persona in the system prompt. Put data and user query in the user turn.

  • Structured output: ask for JSON with a specific schema, validate with Pydantic. Never parse LLM prose with regex — use structured output.

APIs Review

  • Chat Completions: client.chat.completions.create(model, messages) — the core API. Every AI feature starts here.

  • Streaming: stream=True returns tokens as they generate — use for any UI that shows a response being written

  • Function calling: define tools as JSON schemas, the model returns structured tool call arguments — the right way to extract structured data

  • Embeddings: client.embeddings.create(model, input) — returns a vector. Use for semantic search, classification, and similarity scoring.

  • Rate limits and retries: implement exponential backoff on 429 errors, log all API errors, set per-user request quotas to prevent cost spikes

Coding Assistants Review

  • GitHub Copilot: inline suggestions and tab-complete. Best for: boilerplate, well-known patterns, test generation. Weakest on novel architecture.

  • Claude/ChatGPT for architecture: describe the problem, constraints, and tech stack — ask for multiple approaches with tradeoffs, then choose

  • Context matters: paste the relevant function/file alongside the question. Assistants with no context give generic answers.

  • Verify generated code: LLMs produce plausible-looking but incorrect code. Test it. Read it. Especially check error handling, edge cases, and security.

  • Commit habit: generate code → test → read and understand → then commit. 'I don't know why it works' is a liability.

RAG Review

  • Retrieve: embed the query, find top-k nearest chunks in a vector index, return with similarity scores

  • Augment: insert retrieved chunks into the LLM prompt as grounding context alongside the user query

  • Generate: LLM reads context and produces a grounded, cited answer — not a hallucinated one from training weights

  • Chunking: 512 tokens with 50-token overlap is a strong default. Measure retrieval quality before and after changing chunk size.

  • Vector stores: pgvector for Postgres-native apps (ACID, SQL joins), FAISS for local development, Pinecone for hosted scale. Choose based on operational complexity tolerance, not hype.
